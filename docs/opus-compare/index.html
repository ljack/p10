Five frontier AI models build the same todo app. Here's what actually differs.

Same prompt, same tools, same sandbox — Opus 4.6, Opus 4.7, Sonnet 4.6, GPT-5.3-codex, and GPT-5.4 each produced a Todo REST API + a vanilla-JS UI, then passed through black-box E2E tests, objective axe-core a11y checks, a neutral-judge design rubric, and a self-fix pass. Every number here is measured; every artifact is in the repo.

Methodology

A reproducible harness, one prompt per artifact, one model variable, everything else fixed:

  1. API build — Express + Jest/supertest Todo service. Prompt.
  2. Shared CORS patch — identical 6-line middleware added to every API so browser UIs can call it (this plugs a spec hole found by E2E — see §below).
  3. UI build — vanilla HTML/CSS/JS Todo UI with a required data-testid contract. Prompt.
  4. Functional E2E — 12 Playwright tests per build (60 total) exercising every behaviour in the spec.
  5. Objective design analysis — axe-core + CSS/HTML metrics on the live page. Fully deterministic.
  6. LLM design rubric — gpt-5.4 judges each build against 21 dimensions (Nielsen 10 + Gestalt 3 + hierarchy + typography + 3 states + 3 visual a11y). Rubric.
  7. Fix pass — each model receives its own axe-report.json and is asked to fix the violation minimally without breaking the E2E contract. Fix prompt.
  8. Re-verify — rerun E2E + axe. All builds stayed green; all reached 0 axe violations.

Each model ran via pi in JSON mode with --no-context-files --no-extensions --no-skills --no-prompt-templates and a fresh working directory. No AGENTS.md nudging anyone.

A note on the CORS hole. E2E immediately exposed that none of the API builds set CORS headers — the spec didn't demand it. All five models produced the same bug. Rather than pretend each one built a browser-ready API, I applied one shared patch and recorded two verdicts (pre-fix 0/60 and post-fix 60/60). The same E2E harness that catches models also catches your own spec gaps.

Two stable patterns across both tasks:

  • Anthropic faster, OpenAI cheaper. Opus builds in half the wall-clock. Codex models spend more tokens (more thinking, more output) but are priced low enough to come out similar on total cost.
  • Opus-4-7 caches aggressively — input tokens drop to near-zero after turn 1, which is visible in the fix-pass numbers and consistent with the round-1 Todo API experiment.

Functional E2E — 60/60

The Playwright suite runs 12 behavioural tests against each build. All pass everywhere once CORS is in place.

  • Health dot turns ok when API is up
  • Empty-state visible with zero todos
  • Add valid todo appears in list
  • Empty title shows inline error, does not POST
  • Toggle completed updates data-completed and applies strikethrough
  • Inline edit saves on Enter
  • Inline edit cancels on Escape
  • Delete removes the item from DOM
  • Sort: newest first
  • API base change persists across reload
  • Relative time renders
  • No console errors on load + basic interaction

This is the most boring-yet-important finding: with a testid contract in the prompt, every contemporary frontier model produces functionally equivalent UIs. The interesting deltas are everywhere except correctness.

Shared strengths (every model): aesthetic minimalism (5/5 across the board), clean alignment, plain language, readable hierarchy.

Judge-bias disclosure: the design judge is gpt-5.4, which is also a contestant. It scored itself highest. A fairer rerun would use a panel (e.g. Opus + GPT + Gemini) or a human-rated sample. Treat these numbers as directional.

Observations

  1. Different failure modes, not different competence. Every model produced a working, tested app. The real deltas are in defaults — colour palette, landmark choices, typographic scale.
  2. Testable contracts win. Add data-testid + fixed text to the prompt. One E2E suite then audits every model identically.
  3. Codex defaults are terser. GPT-5.3-codex produced the smallest font-size palette and the only zero-violation UI. Coding-tuned models appear to pick more conservative design defaults.
  4. Cache behaviour is a first-class metric. On the fix pass, opus-4-7 and sonnet-4-6 paid near-zero input cost thanks to prompt cache hits; opus-4-6 paid for 16.5k uncached input on the same task.
  5. Spec holes are model-agnostic. The CORS gap hit all five models identically. E2E finds spec bugs unit tests can't.
  6. Judge-bias is measurable. gpt-5.4 rated gpt-5.4 highest — expected. Worth reruning with a panel.

Reproduce in ~25 minutes

Everything — prompts, harness scripts, raw JSONL streams, axe reports, design-rubric JSON, screenshots — is in the repo. Running one model end-to-end:

cd agents-test/opus-compare

# API + UI build + CORS patch + screenshots + objective axe
./scripts/run-full.sh openai/gpt-5.4 gpt-5-4 4141 9191

# Functional E2E (add a project entry first)
cd tests && npx playwright test --project=gpt-5-4

# LLM design rubric
node design/judge.mjs gpt-5-4 reports/gpt-5-4 runs/ui-gpt-5-4

# Fix pass
./scripts/run-fix.sh openai/gpt-5.4 runs/ui-gpt-5-4 prompt/fix-axe.prompt.md runs/ui-gpt-5-4/fix-stream.jsonl

The harness is model-agnostic; any <provider>/<model> that works through pi will run.

Caveats

  • n=1 per model. Variance in LLM runs is real; 3–5 replications needed for claims to mean anything statistically.
  • Judge self-bias. gpt-5.4 is both contestant and judge. Rerun with Opus or a panel.
  • Single task type. Small Todo CRUD. Harder tasks (multi-file refactors, ambiguous bugs, API integration) would likely re-rank things.
  • GPT-5.4-codex not available. The codex fine-tune advertised in pi's model registry returns 404 on the OpenAI API. Substituted with gpt-5.3-codex.
  • Rubric is one opinion. Nielsen + Gestalt is classic but not canonical. Rerun with Cooper's IxD or a heuristic-inquiry rubric for cross-validation.
  • No mobile / no performance. Desktop-only, no TTFB/FCP metrics.
